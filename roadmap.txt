App name: TransacScope

Title: Role-Based Business Finance and Transaction Management

Concept:  
A system to record, track, and report business transactions for a company (e.g., Purchases, Expenses, Income). Users can add, update, request deletion, and query expense and income transactions. Reports summarize daily, weekly, or monthly activity for decision-making. The system also includes email verification during user registration to ensure secure and legitimate access.


How it covers everything:
The project uses PostgreSQL to store expense data, including details like amounts, dates, categories, and descriptions, as well as optional tables for users and expense categories. Using FastAPI and data modeling, expenses, categories, and optionally users are defined as models with Pydantic schemas to validate incoming data and shape responses. PostgreSQL and SQL serve to perform all database operations, such as creating, reading, updating, and deleting expenses, and to query aggregated data for reporting. The system exposes multiple RESTful API endpoints to handle expenses, categories, and report generation, following proper HTTP methods and response codes. As a full stack exercise, the FastAPI backend is fully functional, and endpoints can be tested through tools like Postman or a React.TSX/HTML/CSS frontend. For an extra challenge, the system can implement JWT-based authentication to manage user access, generate downloadable reports, and provide summary endpoints for analytics.


Example scenario: Real-World Inspiration of how the system gonna work

Let’s assume your project is inspired by a computer café (internet/gaming café) business:
-- This TransacScope → tracks all business expenses, independent of sales.
This includes daily operational costs (electricity, internet), hardware and equipment purchases (PCs, peripherals), maintenance and repairs, employee salaries, rent, food and beverage inventory for resale inside the café, and advertising and promotional expenses (such as social media promotions and outdoor banners).


Expense categories for the business:
Operating Overhead:  miscellaneous expenses (rent, ELectricity, Internet, Water) // daily cost
Inventory & Supplies: Consumables and short-term items (food & beverages for resale, cleaning supplies, stock peripherals)
Salary: Payments to employees (technicians, staff, attendants)
Equipment & Assets: Long-term investments (new computers, upgrades, gaming chairs, routers/switches, doors)
Marketing & Advertising: (Social media ads, outdoor banners, promotional events)
Other Business Expense: Customizable by super admin and admin. they can amange all categories


Income categories for the business:
Standard Room Usage: regular computer rental income (hourly gaming, internet browsing)
VIP Room Usage: premium room rental income (private room access, high-end PC usage)
Digital Top-Up / Game Load: commission-based digital sales (game credits, prepaid load, e-wallet top-ups)
Food & Beverage: in-store product sales (snacks, instant meals, soft drinks)
Tournaments: event-based gaming revenue (registration fees, sponsorship shares)
Other Business Income: Customizable by super admin and admin. they can amange all categories


Workflow with users and admin: 
Roles are fixed for demo: 1 Super Admin, 1 Admin, 5 Standard Users (possibly1 standard user per categories, but no stricly).

--- Super Admin (Business Owner): Can CRUD categories, promote and demote users, soft delete users, view all transactions or their own, delete any transactions, edit their own transactions, see soft-deleted transactions, and generate reports for all or own transactions.

-- Admin (Manager): Can CRUD categories, view all transactions or their own, soft delete transactions, edit their own transactions, see soft-deleted transactions, generate reports for all or own transactions, and manage deletion requests from Standard users.

- Standard Users (Staff): Can create, view, and update their own transactions, request deletion of their transactions, generate and view reports for their own transactions, and view the edit/delete history of their own transactions.

* The key difference between Admin and Super Admin is that the Super Admin can manage users fully, including promoting, demoting, and soft deleting them, while the Admin cannot.


Super/Admin side bar:
Dashboard: Full overview of all users and transactions
Manage Categories - add edit delete categories
Transactions - View all transactions, add new, edit own, soft delete
Reports - Generate reports for all users or their own transactions
Manage Users / Roles: Approve or decline deletion requests, promote/demote users (Super Admin only)

standard user side bar:
Dashboard: Overview of their own transactions only
Transactions: Create, view, and update their own transactions, request deletion
Reports / Summary: Generate and view reports for their own transactions only


STACKS: 
Database:PostgreSQL 
- Store all transaction data: amount, date, type (Expense/Income), category, description.
- Store related tables: users (if authentication is implemented), transaction categories, reports history, log history, emai_verification
- Can include relational constraints: e.g., each transaction belongs to a category, optional user table.


FastAPI & Data Modeling: 
- Models for transactions, categories, and optionally users.
- Define Pydantic schemas for request validation and response formatting.
- Include relationships between models (e.g., transactions → categories).
- Email verifications


SQL scripts with PostgreSQL: 
- layer to interact with PostgreSQL.
- CRUD operations: Create, Read, Update, Delete transactions.
- CRUD Categories and manage users for admins
- Query filtering for reports (e.g., sum totals per category or per time period, total income vs total expense, net profit per period).


API / REST: 
Endpoints for:
/transactions → list, create, update, delete transactions. ((expense or income).)
/categories → manage categories.
/reports → generate reports (daily, monthly, etc., including income vs expense summaries), have downloadable for PDF.
- Use REST principles: proper HTTP methods (GET, POST, PUT, DELETE).

Full Stack: FastAPI serves as backend.
- Configured environment includes: FastAPI app, PostgreSQL, SQL, Pydantic models, API routes.
